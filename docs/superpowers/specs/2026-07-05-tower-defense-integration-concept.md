# 塔防融合构思：长夜守城

日期：2026-07-05

## 一句话定位

塔防不是独立小游戏，而是《造物者考核3D》的“守护层”：玩家在主玩法里创造、救援、稳定区域；塔防负责验证这些成果能不能在裂隙反扑中守住。

## 放在什么地方

推荐放在“第六天：终考”和“第七天：裂隙空域”之间。

当前主线已经有“终考：第七天之前”。塔防可以承接这个名字：第七天还没到，世界必须先熬过最后一夜。玩家之前创造过的造物、救下的居民、稳定过的区域，会在这一夜变成防线的一部分。

建议命名：

- 长夜守城
- 裂隙防线
- 守夜模式
- 第七天之前：最后防线

## 核心叙事理由

造物者不能只会“创造奇迹”，还要证明奇迹能被保存下来。塔防关卡代表裂隙对已稳定区域的反扑：那些被救下的居民、临时修复的道路、被安抚的灾害，都会在长夜里接受压力测试。

## 采用范围

第一阶段不建议把塔防游戏完整搬成主流程的大系统。更稳的做法是先采纳它的“无尽/波次防守”能力，做成短流程插入玩法。

优先采用：

- 波次生成
- 防御建筑/塔升级
- 无尽模式
- 基地生命值
- 敌人路线和压力递增

暂缓采用：

- 完整塔防剧情
- 大量原有关卡配置
- 复杂资源线
- 与主游戏重复的成长系统

## 与主游戏怎么连接

主游戏传入塔防的上下文：

```js
{
  mode: 'night_watch',
  day: 6,
  regionId,
  regionTitle,
  entropy,
  rescuedResidents,
  lostResidents,
  recentCreations,
  discoveredLore,
  playerStyle
}
```

塔防回传主游戏的结果：

```js
{
  survivedWaves,
  baseDamage,
  residentsProtected,
  residentsInjured,
  entropyDelta,
  unlockedMaterials,
  notableMoment
}
```

回传结果进入 `WorldSimulation` / `EventBus`，再影响后续第七天、居民记忆、世界传说和结局文本。

## 玩法映射

| 主游戏元素 | 塔防表现 |
|---|---|
| 自然语言造物 | 防御塔、屏障、陷阱、区域技能 |
| 救下的居民 | 修理员、炮手、临时增益、剧情保护对象 |
| 熵值 | 敌人数量、速度、变异概率 |
| 区域地形 | 塔防路线主题和障碍分布 |
| AI Storyteller | 波次事件、突袭、喘息窗口 |
| 造物副作用 | 塔过载、误伤、腐化、资源消耗 |
| 失败记录 | 特殊敌人或负面波次 |

## 模式设计

### 守夜短关

主线插入模式，1 到 3 波即可结束。目标是让塔防成为节奏变化，而不是打断主流程。

用途：

- 区域刚被拯救后，确认是否稳固
- 重要居民迁移时，守住撤离路线
- 第七天之前，进行一次最终守城

### 长夜危机

中等长度，3 到 5 波。用于主线高潮前，压力更大，结果会明显影响后续难度。

用途：

- 第六天终考后的固定节点
- 高熵区域触发的惩罚关
- 玩家主动选择“守住这个地方”时开启

### 无尽守夜

通关后或支线开放。无尽模式可以作为塔防原作最完整的保留点。

用途：

- 刷材料
- 测试造物组合
- 让 AI 根据玩家历史不断变换波次
- 作为第七天之后的长线玩法

## AI 驱动点

塔防不需要每一波都请求 AI。更合适的是：AI 负责生成“防守主题”和“特殊事件候选”，本地规则负责真正执行。

可生成内容：

- 本次守夜主题：洪水反扑、记忆迷雾、失语战争余波
- 特殊波次名与描述
- Boss 或精英敌人的叙事身份
- 居民通讯文本
- 防守后的传说文本

必须本地规则控制：

- 敌人数值
- 波次数
- 塔伤害
- 资源产出
- 胜败结果
- 回写世界状态

## 技术接入方式

塔防目前是单独 HTML，因此第一阶段推荐“低耦合接入”：

1. 把 HTML 和资源放到 `public/modes/tower-defense/`。
2. 先保持它能独立运行。
3. 增加一个桥接脚本 `towerBridge.js`，通过 URL 参数、`localStorage` 或 `postMessage` 接收主游戏上下文。
4. 塔防结束后把结果写回同一个桥接通道。
5. 主游戏读取结果，并转成世界事件。

建议目录：

```text
public/modes/tower-defense/
  index.html
  towerBridge.js
  assets/
```

## 第一阶段验收

- 能从主游戏打开塔防模式。
- 塔防能读取至少 3 个主游戏状态：`entropy`、`rescuedResidents`、`recentCreations`。
- 至少 1 个主游戏造物能映射成塔或技能。
- 塔防结束后能回传 `survivedWaves` 和 `entropyDelta`。
- 主游戏日志里能记录一次 `defense_resolved` 世界事件。

## 关键风险

- 玩法膨胀：塔防、空战和主策略都很完整，必须让塔防先做“短插入”和“无尽保留”，不要抢主流程。
- 状态双写：塔防不能直接改主世界，只能回传结果，由主游戏统一写入 `EventBus`。
- 美术和 UI 风格不一致：第一阶段可以接受，但进入主流程前需要统一入口、结算页和文本语气。
- 原 HTML 结构未知：等拿到网页源码后再决定是包 iframe、直接迁移，还是抽出游戏循环。

## 暂定优先级

高。塔防和当前项目的“救援、守护、居民、区域稳定”天然贴合，适合先于飞机大战接入，作为第七天之前的防线玩法。
