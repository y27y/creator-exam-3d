# AGENTS.md - Agent 工作约定

> 本文件记录 AI agent 在本仓库中的协作流程。核心目标是：每次开工先及时 get 上下文，明确我们真正要交付什么，并让每一步变更都方便版本管理、回看和回滚。

## 每次任务开始必须先 get

- 先执行 `git status`，确认当前分支、已修改文件、未跟踪文件。
- 必要时执行 `git diff` / `git log`，获取当前改动范围和最近版本脉络。
- 阅读本文件、`README.md`、`CLAUDE.md` 以及任务相关源码/文档，不凭记忆直接改。
- 发现已有用户改动时，默认保留并协作，不擅自覆盖、回退或重写。

## 先明确我们需要的东西

在动手前，agent 需要把任务拆成可验证目标：

- **目标**：这次到底要做什么，是修 bug、加功能、改体验、写文档，还是做概念设计。
- **交付物**：最终需要产出哪些东西，例如代码、测试、文档、截图、运行结果、PR 或提交。
- **验收标准**：什么现象说明完成了，例如测试通过、页面可操作、接口返回正确、文档补齐。
- **范围边界**：哪些文件/模块应该动，哪些无关改动不要碰。
- **版本节点**：完成到哪个小阶段需要检查 diff、记录结果，必要时提交。

如果这些信息能从上下文推断，agent 直接推进；如果缺少关键条件且会影响方向，先向用户确认。

## 标准推进流程

1. **Get 上下文**：查看 git 状态、读相关文件、理解当前实现。
2. **确认目标**：用一句话明确本轮要交付的结果。
3. **小步修改**：优先做最小可行改动，沿用项目现有结构、命名和风格。
4. **同步测试**：涉及逻辑时补充或运行 `debug/` 相关测试；涉及文档时同步更新 `docs/`。
5. **检查版本**：阶段完成后查看 `git diff` / `git status`，确认只包含本轮相关改动。
6. **交付说明**：说明改了什么、怎么验证、还有哪些风险或后续事项。

## 版本管理要求

- 每完成一个可验证的小阶段，及时检查变更并记录关键结果。
- 需要提交时按功能边界拆分提交，避免把无关改动混在同一次版本记录里。
- 不把临时调试文件、无关格式化、环境噪音混进版本。
- 如果发现工作区已有无关改动，只报告或避开，不主动清理。

## 开发协作要求

- 优先沿用本项目现有架构、工具链和测试方式。
- 修改核心机制时，同步考虑 `README.md`、`CLAUDE.md`、`docs/systems/architecture.md` 是否需要更新。
- 新增系统或重要行为时，优先补测试；不能测试时要说明原因。
- 交付前尽量运行相关测试，并把验证结果写清楚。

## 视觉交付要求

- 所有新增或修改的视觉元素都必须放回完整画面中评估构图、层级、明度、材质和风格一致性，不能只验证局部对象或单张源图。
- 不使用默认辅助线、默认材质、临时圆圈、占位几何或单层色块作为最终效果；重要地形、模型和特效需要有可辨识的结构、材质层次与受光关系。
- 代码生成的线条和边界必须有明确视觉语义，并与关卡主题协调；图片资源不得烘焙会和实时棋盘冲突的格线、UI、圆环或硬阴影。
- 涉及 Three.js、Canvas、图片或响应式界面的改动，交付前必须查看桌面与移动端实机截图，检查非空渲染、遮挡、文字可读性和动态内容布局。
- 图片生成工具不可用时，先完成不依赖新图的代码与材质工作，并提供可直接生成、转换、替换和验收的完整生产规格；不得声称生成了实际不存在的资源。
